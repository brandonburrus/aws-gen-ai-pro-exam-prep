## 1. AWS X-Ray

### 1.1 Distributed Tracing Overview

AWS X-Ray collects data about requests as they flow through distributed applications. It
builds a visual **service map** showing how services connect and where bottlenecks occur.

**Core Concepts:**

| Concept | Definition | Exam Relevance |
|---------|-----------|----------------|
| **Segment** | Data about work done by a single compute resource (hostname, request/response details, timing, errors) | Each service produces one segment per traced request |
| **Subsegment** | Granular breakdown within a segment (downstream AWS SDK calls, HTTP calls, SQL queries) | Used to trace Bedrock API calls from Lambda |
| **Trace** | Collection of all segments generated by a single request as it crosses services | Trace ID propagated via `X-Amzn-Trace-Id` header |
| **Service Map** | Visual graph of nodes (services) and edges (connections) generated from trace data | Shows full topology of your AI application |
| **Sampling** | Algorithm controlling which requests are traced (default: 1st request/second + 5% of additional) | Configurable via sampling rules; prevents cost overrun |
| **Annotations** | Key-value pairs (string/number/boolean) **indexed** for search and filter expressions | Up to 50 per trace; use for filtering AI traces |
| **Metadata** | Key-value pairs of any type, **not indexed** | Store prompt/response payloads, model config |
| **Groups** | Named filter expressions that partition traces | Create groups for AI vs. non-AI traffic |

**Tracing Header Format:**
```
X-Amzn-Trace-Id: Root=1-5759e988-bd862e3fe1be46a994272793;Parent=53995c3f42cd8ad8;Sampled=1
```

**Service Map Data Retention:** 30 days.

**Segment Document Size Limit:** 64 KB.

### 1.2 Tracing Bedrock API Calls

X-Ray integrates with Amazon Bedrock AgentCore for distributed tracing of AI agent
invocations. When the AWS SDK makes Bedrock API calls from an instrumented service (e.g.,
Lambda), X-Ray captures these as subsegments automatically.

**How it works:**

1. Lambda function instrumented with X-Ray creates a root segment
2. AWS SDK call to `bedrock-runtime` (e.g., `InvokeModel`, `Converse`) generates a subsegment
3. Subsegment records: latency, status code, request/response size, errors
4. If the downstream service is instrumented, its segment replaces the inferred segment

**Amazon Bedrock AgentCore + X-Ray Integration:**
- AgentCore supports trace context propagation via the `X-Amzn-Trace-Id` header
- Custom instrumentation via the AWS Distro for OpenTelemetry (ADOT) SDK
- Requires enabling CloudWatch Transaction Search (one-time account setup)
- Track requests across agent -> tool -> knowledge base -> FM boundaries

**Bedrock-Specific Observability (Complementary to X-Ray):**

| Source | What it Captures |
|--------|-----------------|
| Bedrock Agent Trace | Orchestration steps, tool calls, RAG retrieval |
| Model Invocation Logging | Full prompt/response, token counts, model ID |
| CloudWatch Metrics | Invocation count, latency, error rate |
| X-Ray | End-to-end latency, service dependency map |

### 1.3 Service Maps for AI Applications

A typical GenAI service map shows:

```
Client -> API Gateway -> Lambda (preprocessing)
                           -> Bedrock Runtime (inference)
                           -> DynamoDB (session store)
                           -> S3 (knowledge base source)
```

X-Ray generates **inferred segments** for services that do not send their own segments
(e.g., DynamoDB, S3). These appear as nodes on the service map based on the subsegment data
from the calling service.

**Exam Tip:** The service map shows health (error/fault rates) and latency at each node.
Use this to identify which component in your AI pipeline is the bottleneck (e.g., Bedrock
inference latency vs. vector database retrieval time).

### 1.4 Segments and Subsegments in Practice

**Lambda + X-Ray:**
- When active tracing is enabled on a Lambda function, Lambda creates the root segment
  automatically
- Your code can only create subsegments within Lambda (not new root segments)
- Lambda adds two nodes to the trace map: one for the Lambda service, one for the function

```typescript
// Node.js X-Ray SDK example in Lambda
import AWSXRay from 'aws-xray-sdk';
import { BedrockRuntimeClient, ConverseCommand } from '@aws-sdk/client-bedrock-runtime';

// Wrap the AWS SDK client for automatic subsegment creation
const client = AWSXRay.captureAWSv3Client(
  new BedrockRuntimeClient({ region: 'us-east-1' })
);

export const handler = async (event) => {
  // Bedrock call is automatically captured as a subsegment
  const response = await client.send(new ConverseCommand({
    modelId: 'anthropic.claude-3-haiku-20240307-v1:0',
    messages: [{ role: 'user', content: [{ text: event.prompt }] }],
  }));

  // Add custom annotation (indexed, searchable)
  const segment = AWSXRay.getSegment();
  const subsegment = segment.addNewSubsegment('ProcessResponse');
  subsegment.addAnnotation('modelId', 'claude-3-haiku');
  subsegment.addAnnotation('tokenCount', response.usage?.totalTokens ?? 0);

  // Add metadata (not indexed, stores complex objects)
  subsegment.addMetadata('prompt', event.prompt, 'ai');
  subsegment.addMetadata('response', response.output, 'ai');
  subsegment.close();

  return response.output;
};
```

### 1.5 Integration with Lambda and API Gateway

**Lambda:**
- Enable active tracing in function configuration (`TracingConfig: { Mode: Active }`)
- Lambda runs the X-Ray daemon on Java and Node.js runtimes
- Creates facade segment automatically; your code adds subsegments

**API Gateway:**
- Supports active tracing (enable per stage)
- Adds a node for the gateway stage to the service map
- Propagates trace header downstream to Lambda/other integrations

**Exam Gotcha:** API Gateway uses X-Ray sampling rules to decide which requests to trace.
Lambda uses passive instrumentation when called by API Gateway (respects the upstream
sampling decision).

### 1.6 Annotations and Metadata for AI Traces

| Feature | Annotations | Metadata |
|---------|------------|----------|
| Indexed | Yes | No |
| Searchable via filter expressions | Yes | No |
| Value types | String, Number, Boolean | Any (objects, arrays, etc.) |
| Limit | 50 per trace | No hard limit (within 64 KB segment size) |
| Key limit | 500 alphanumeric characters | No restriction |
| Value limit | 1,000 Unicode characters | Serializable to JSON |

**GenAI-Specific Annotation Strategy:**

```
annotation["modelId"] = "anthropic.claude-3-haiku"
annotation["useCase"] = "summarization"
annotation["guardrailApplied"] = true
annotation["tokenBudgetExceeded"] = false
```

**GenAI-Specific Metadata Strategy:**

```json
{
  "namespace": "ai",
  "systemPrompt": "You are a helpful assistant...",
  "userPrompt": "Summarize this document...",
  "modelResponse": "The document covers...",
  "inferenceConfig": { "temperature": 0.7, "maxTokens": 1024 }
}
```

**Filter Expression Examples:**
```
annotation.modelId = "claude-3-haiku"
annotation.tokenCount > 1000
annotation.useCase = "rag-query" AND responsetime > 5
```

### 1.7 Prompt Observability Pipelines (Task 5.2.5)

A prompt observability pipeline captures, stores, and analyzes all prompts and responses
flowing through your GenAI application.

**Architecture:**

```
User Request
  -> API Gateway (X-Ray trace starts)
    -> Lambda (preprocessing, prompt assembly)
      -> Bedrock (inference, model invocation logging enabled)
        -> Model invocation logs -> S3 -> Athena (analytics)
        -> CloudWatch Logs (structured logging with embedded metrics)
        -> X-Ray subsegments (latency, error tracking)
    -> Post-processing Lambda
      -> CloudWatch custom metrics (token usage, hallucination rate)
      -> OpenSearch (semantic search over responses)
```

**Key Metrics to Monitor:**

| Metric | Source | Why |
|--------|--------|-----|
| Inference latency per model | X-Ray subsegments | Detect slowdowns |
| Token usage (input + output) | Bedrock invocation logs | Cost control, detect prompt bloat |
| Error rate / fault rate | X-Ray service map | Pipeline stability |
| Hallucination rate | Custom metric (post-processing) | Output quality |
| Guardrail intervention rate | Bedrock guardrail logs | Content safety |
| Tool selection accuracy | Bedrock agent trace | Agent behavior alignment |
| Cost per session | CloudWatch embedded metrics | FinOps |

**Exam Tip:** The exam expects you to know that X-Ray handles cross-service latency and
dependency mapping, while Bedrock model invocation logging captures prompt/response content
and token usage. These are complementary, not competing, observability mechanisms.

**Migration Note:** The X-Ray SDK enters maintenance mode on February 25, 2026. AWS
recommends migrating to OpenTelemetry via ADOT (AWS Distro for OpenTelemetry). The exam may
reference both approaches.

## 2. AWS Amplify

### 2.1 Overview and Frontend Framework Integration

AWS Amplify is a full-stack development platform for building web and mobile applications.
Amplify Gen 2 uses a TypeScript-first, code-based approach for defining backends.

**Supported Frameworks:**
- React, Next.js, Vue, Angular, React Native, Flutter, Android, Swift

**Core Components:**
- `@aws-amplify/backend` - Backend definition (data, auth, storage, functions)
- `@aws-amplify/ui-react` - Pre-built UI components
- `amplify sandbox` - Local cloud development environment
- `amplify deploy` - CI/CD deployment

### 2.2 Amplify AI Kit -- AI-Powered UI Components (Task 2.5.2)

The Amplify AI Kit is the primary exam topic for Amplify. It provides declarative UI
components and backend infrastructure for building AI-powered applications with Amazon
Bedrock.

**Two Route Types:**

| Route Type | Pattern | API Style | Storage | Use Case |
|-----------|---------|-----------|---------|----------|
| **Conversation** | `a.conversation()` | Async, multi-turn, streaming | Messages stored in DynamoDB | Chat UIs, conversational search |
| **Generation** | `a.generation()` | Sync, single request-response | No automatic storage | Summarization, structured data extraction, alt-text |

**Cloud Infrastructure Created by AI Kit:**

```
AppSync (GraphQL API)
  -> Lambda (conversation handler)
    -> Bedrock (Converse API)
  -> DynamoDB (conversation history)
```

- **AWS AppSync**: Serverless GraphQL API layer; handles authorization and real-time
  subscriptions for streaming responses
- **AWS Lambda**: Executes conversation logic; retrieves history and invokes Bedrock
- **Amazon DynamoDB**: Stores conversation and message data; conversations scoped per user
- **Amazon Bedrock**: Foundation model inference via the Converse API

**Defining AI Routes in Amplify:**

```typescript
// amplify/data/resource.ts
import { a, defineData, type ClientSchema } from '@aws-amplify/backend';

const schema = a.schema({
  // Multi-turn conversation route
  chat: a.conversation({
    aiModel: a.ai.model('Claude 3.5 Haiku'),
    systemPrompt: 'You are a helpful assistant',
  })
  .authorization((allow) => allow.owner()),

  // Single-shot generation route
  generateRecipe: a.generation({
    aiModel: a.ai.model('Claude 3.5 Haiku'),
    systemPrompt: 'You are a helpful assistant that generates recipes.',
  })
  .arguments({
    description: a.string(),
  })
  .returns(
    a.customType({
      name: a.string(),
      ingredients: a.string().array(),
      instructions: a.string(),
    })
  )
  .authorization((allow) => allow.authenticated()),
});
```

**Amplify AI Kit uses Bedrock's Converse API** -- this is the unified API that works
across models. The kit handles:
- Conversation history management
- Tool configuration and invocation
- Real-time subscriptions for streaming
- System prompt injection

**Pre-built `<AIConversation>` Component:**
- Themed and extensible React component
- Handles real-time subscriptions, chat history, and UI state
- Manages multi-turn invocations automatically

**Supported Models (via Bedrock Converse API):**

| Provider | Models | Conversation | Generation |
|----------|--------|:------------:|:----------:|
| Anthropic | Claude 3 Haiku, 3.5 Haiku | Yes | Yes |
| Amazon | Nova Pro, Nova Lite, Nova Micro | Yes | No |
| AI21 Labs | Jurassic-2 Large, Mini | Yes | No |
| Cohere | Command R, Command R+ | Yes | No |
| Meta | Llama 3 models | Yes | No |

**Prompt Structure in Amplify AI Kit:**
1. **System prompt** -- High-level instructions (required for all routes)
2. **Messages** -- Conversation history (managed automatically)
3. **Tool configuration** -- Tools the model can invoke (managed automatically)

### 2.3 Backend Integration with Bedrock

Beyond the AI Kit, you can connect Amplify backends directly to Bedrock as an HTTP data
source using custom resolvers:

```typescript
// amplify/data/resource.ts -- Custom Bedrock data source
const bedrockDataSource = defineData({
  schema,
  authorizationModes: { defaultAuthorizationMode: 'userPool' },
});

// amplify/data/bedrock.js -- Custom resolver for Bedrock
export function request(ctx) {
  return {
    method: 'POST',
    resourcePath: '/model/anthropic.claude-3-haiku-20240307-v1:0/invoke',
    params: {
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        anthropic_version: 'bedrock-2023-05-31',
        messages: [{ role: 'user', content: ctx.args.prompt }],
        max_tokens: 1024,
      }),
    },
  };
}

export function response(ctx) {
  return JSON.parse(ctx.result.body).content[0].text;
}
```

### 2.4 Authentication for AI Applications

Amplify uses Amazon Cognito for authentication. For AI routes:

- **Conversation routes** use `.authorization((allow) => allow.owner())` -- conversations
  are scoped to the authenticated user
- **Generation routes** typically use `.authorization((allow) => allow.authenticated())`
- The user's OIDC access token is passed from the client to AppSync, which authorizes the
  request before forwarding it to Lambda/Bedrock

**Exam Gotcha:** Amplify AI Kit conversation routes store messages in DynamoDB with
owner-based authorization. Each user can only see their own conversation history. This is
built into the architecture, not something you configure manually.

**Exam Gotcha:** Amplify AI Kit uses the Bedrock **Converse** API (not InvokeModel). The
Converse API provides a unified interface across models, which is why Amplify can support
model swapping without code changes.

## 3. CI/CD for GenAI (CodePipeline, CodeBuild, CodeDeploy)

### 3.1 CI/CD Pipeline Architecture for AI Applications (Task 2.3.5)

GenAI CI/CD differs from traditional software CI/CD because you must version and test
**prompts, agent configurations, and model selections** alongside code and infrastructure.

**Typical CI/CD Workflow for Serverless AI:**

| Stage | Actions | Tooling |
|-------|---------|---------|
| 1. Code + Prompt Commit | Push Lambda code, CDK stacks, prompt files to Git | GitHub, GitLab, CodeCommit |
| 2. Build + Lint | Validate syntax, prompt format, schema alignment | CodeBuild, ESLint, Black, yamllint |
| 3. Unit Tests + Prompt Regression | Run logic tests and golden prompt-response tests | CodeBuild, pytest, promptfoo |
| 4. IaC Validation | Synthesize and validate CDK/CloudFormation templates | `cdk synth`, `cfn-lint` |
| 5. Integration Test | Deploy to staging, invoke full AI workflow | CodeBuild, mocked agents |
| 6. Manual/Auto Approval | Review model cost impact, prompt change checklist | CodePipeline approval gates |
| 7. Deploy to Production | Promote stacks, update Bedrock agent configs | CodeDeploy, CDK, SAM CLI |
| 8. Post-Deploy Smoke Test | Validate agent outputs, log capture, rollback readiness | CloudWatch Synthetics |
| 9. Monitor + Observe | Dashboards, cost alerts, token usage monitors | CloudWatch, X-Ray, Bedrock logs |

### 3.2 AWS CodePipeline

CodePipeline orchestrates the entire release process as a series of stages and actions.

**Key Features for GenAI:**
- **Pipeline stages**: Source -> Build -> Test -> Approval -> Deploy
- **Source integrations**: GitHub, CodeCommit, S3, ECR
- **Parallel actions**: Run multiple CodeBuild projects in parallel within a stage
- **Manual approval gates**: Require human review before deploying prompt changes to production
- **Declarative templates**: Define pipelines as CloudFormation/CDK

**GenAI-Specific Pipeline Pattern:**

```
Source (Git)
  -> Build Stage
     -> CodeBuild: Lint + unit tests
     -> CodeBuild: Prompt regression tests (parallel)
     -> CodeBuild: cdk synth + cfn-lint (parallel)
  -> Staging Deploy Stage
     -> CloudFormation deploy to staging
     -> CodeBuild: Integration tests against staging
  -> Approval Gate
     -> Manual review of prompt changes + cost impact
  -> Production Deploy Stage
     -> CodeDeploy: Blue/green or canary deployment
  -> Post-Deploy Stage
     -> CloudWatch Synthetics canary tests
```

### 3.3 AWS CodeBuild

CodeBuild is a fully managed build service that compiles code, runs tests, and produces
deployment artifacts.

**Key Features for GenAI:**
- **Custom build environments**: Docker images with ML/AI dependencies
- **Parallel test execution**: Shard test suites across multiple build nodes using
  `codebuild-tests-run` utility
- **Test report groups**: Consolidate results from parallel test runs
- **Buildspec.yml**: Define build commands, phases, and artifacts

**Example Buildspec for GenAI Testing:**

```yaml
version: 0.2
phases:
  install:
    runtime-versions:
      nodejs: 20
    commands:
      - npm ci
  pre_build:
    commands:
      - npm run lint
      - npx cdk synth
      - npx cfn-lint cdk.out/*.template.json
  build:
    commands:
      - npm test                         # Unit tests
      - npm run test:prompts             # Prompt regression tests
      - npm run test:security            # Security scanning
  post_build:
    commands:
      - npm run test:integration         # Integration tests
reports:
  prompt-tests:
    files:
      - 'test-results/prompt-regression.xml'
    file-format: JUNITXML
  security-scan:
    files:
      - 'test-results/security-scan.xml'
    file-format: JUNITXML
artifacts:
  files:
    - 'cdk.out/**/*'
    - 'prompts/**/*'
```

### 3.4 Automated Testing for GenAI

**Prompt Regression Testing:**
- Treat prompts as versioned assets in source control (e.g., `/prompts/v1/agent-support-en.yaml`)
- Maintain "golden" test cases: known input -> expected output pairs
- Use tools like `promptfoo` for automated prompt evaluation
- Compare model outputs against expected patterns/scores

**Types of Tests in GenAI CI/CD:**

| Test Type | What It Validates | When |
|-----------|------------------|------|
| Unit tests | Business logic, data transformations | Build stage |
| Prompt regression | System prompt + user input -> expected output pattern | Build stage |
| Schema validation | CDK/CloudFormation template correctness | Build stage |
| Security scan | Dependency vulnerabilities, IAM over-permissions | Build stage |
| Integration tests | Full workflow: API -> Lambda -> Bedrock -> response | Staging deploy |
| Smoke tests | Production endpoint health, basic inference | Post-deploy |

**Deploy Bedrock Agent Updates Only When:**
1. Prompt regression tests pass
2. Tool permissions match IAM templates
3. Confidence thresholds or validation Lambda results meet acceptable criteria

### 3.5 Security Scanning in AI Pipelines

- **Dependency scanning**: Check AI/ML library versions for known vulnerabilities
- **IAM policy analysis**: Validate least-privilege permissions for Bedrock access
- **Prompt injection testing**: Test for prompt injection vulnerabilities
- **Data leakage checks**: Ensure sensitive data is not embedded in prompts or logs
- **cfn-lint + cfn-guard**: Validate CloudFormation templates against security policies

### 3.6 AWS CodeDeploy -- Rollback and Deployment Strategies

CodeDeploy automates deployments to EC2, Lambda, and ECS. For GenAI applications, Lambda
and ECS deployments are most relevant.

**Lambda Deployment Strategies:**

| Strategy | Behavior | Use Case |
|----------|----------|----------|
| `AllAtOnce` | 100% traffic shifted immediately | Non-critical, dev environments |
| `Canary10Percent5Minutes` | 10% for 5 min, then 100% | Quick validation |
| `Canary10Percent30Minutes` | 10% for 30 min, then 100% | Production AI endpoints |
| `Linear10PercentEvery1Minute` | +10% per minute over 10 min | Gradual rollout |
| `Linear10PercentEvery10Minutes` | +10% every 10 min over ~100 min | Conservative rollout |

**How Lambda Canary/Linear Works:**
1. New Lambda version is published
2. CodeDeploy creates a weighted alias pointing to old + new versions
3. Traffic shifts gradually per the deployment configuration
4. Pre-traffic hook Lambda runs validation before any traffic shifts
5. Post-traffic hook Lambda validates after shift completes
6. CloudWatch alarms trigger automatic rollback if errors spike

**ECS Blue/Green Deployments:**
- Two task sets behind a load balancer (blue = current, green = new)
- Traffic shifts via target group weights
- Supports canary and linear strategies (same patterns as Lambda)
- Configurable bake period before retiring the blue environment
- Lifecycle hooks can run Lambda functions for synthetic testing

**SAM Template for Gradual Lambda Deployment:**

```yaml
MyFunction:
  Type: AWS::Serverless::Function
  Properties:
    Handler: index.handler
    Runtime: nodejs20.x
    AutoPublishAlias: live
    DeploymentPreference:
      Type: Canary10Percent5Minutes
      Alarms:
        - !Ref MyAlarm
      Hooks:
        PreTraffic: !Ref PreTrafficHook
        PostTraffic: !Ref PostTrafficHook
```

**Exam Gotcha:** For Lambda, `AutoPublishAlias` must be set before you can use
`DeploymentPreference`. The first deployment must publish the alias, and the second
deployment onward can use gradual traffic shifting.

**Exam Gotcha:** CodeDeploy for Lambda uses **weighted aliases** to split traffic. This is
different from API Gateway canary deployments which use stage canary settings.

## 4. AWS CDK and CloudFormation

### 4.1 Infrastructure as Code for AI Services

AWS CDK (Cloud Development Kit) lets you define cloud infrastructure in TypeScript, Python,
Java, C#, or Go. CDK synthesizes to CloudFormation templates.

**CDK Key Concepts:**

| Concept | Description |
|---------|-------------|
| **App** | Root of the construct tree; contains one or more stacks |
| **Stack** | Unit of deployment; maps 1:1 to a CloudFormation stack |
| **Construct** | Building block; L1 (raw CFN), L2 (opinionated defaults), L3 (patterns) |
| **`cdk synth`** | Synthesizes CDK app to CloudFormation template |
| **`cdk deploy`** | Deploys the synthesized template |
| **`cdk diff`** | Shows what changes would be applied |

**CDK for GenAI Workflow:**
1. Write CDK TypeScript code defining Bedrock agents, Lambda functions, IAM roles
2. `cdk synth` -> generates CloudFormation template
3. `cfn-lint` validates the template
4. `cdk deploy` creates/updates the stack
5. CI/CD pipeline automates steps 2-4

### 4.2 Bedrock Resources in CloudFormation / CDK

CloudFormation provides L1 (low-level) resource types for all Bedrock resources. CDK wraps
these as L1 constructs (prefixed with `Cfn`).

**Key CloudFormation Resource Types:**

| Resource Type | Purpose |
|--------------|---------|
| `AWS::Bedrock::Agent` | Define a Bedrock agent with model, instructions, action groups, knowledge bases |
| `AWS::Bedrock::AgentAlias` | Create an alias for agent versioning and deployment |
| `AWS::Bedrock::KnowledgeBase` | Create a knowledge base for RAG (vector store config, embedding model) |
| `AWS::Bedrock::DataSource` | Connect S3/other data to a knowledge base for ingestion |
| `AWS::Bedrock::Guardrail` | Define content filters, denied topics, sensitive info filters, word filters |
| `AWS::Bedrock::GuardrailVersion` | Version a guardrail configuration |
| `AWS::Bedrock::Flow` | Define a Bedrock Flow (visual prompt chaining) |
| `AWS::Bedrock::FlowVersion` | Version a flow |
| `AWS::Bedrock::FlowAlias` | Alias for a flow version |
| `AWS::Bedrock::Prompt` | Define a reusable prompt template |
| `AWS::Bedrock::PromptVersion` | Version a prompt template |

**CDK Example -- Bedrock Agent with Guardrail:**

```typescript
import * as cdk from 'aws-cdk-lib';
import * as bedrock from 'aws-cdk-lib/aws-bedrock';
import * as iam from 'aws-cdk-lib/aws-iam';

const agentRole = new iam.Role(this, 'AgentRole', {
  assumedBy: new iam.ServicePrincipal('bedrock.amazonaws.com'),
  managedPolicies: [
    iam.ManagedPolicy.fromAwsManagedPolicyName('AmazonBedrockFullAccess'),
  ],
});

const guardrail = new bedrock.CfnGuardrail(this, 'ContentGuardrail', {
  name: 'my-ai-guardrail',
  blockedInputMessaging: 'Your input was blocked by content filters.',
  blockedOutputsMessaging: 'The response was blocked by content filters.',
  contentPolicyConfig: {
    filtersConfig: [
      { type: 'SEXUAL', inputStrength: 'HIGH', outputStrength: 'HIGH' },
      { type: 'VIOLENCE', inputStrength: 'HIGH', outputStrength: 'HIGH' },
      { type: 'HATE', inputStrength: 'HIGH', outputStrength: 'HIGH' },
      { type: 'INSULTS', inputStrength: 'HIGH', outputStrength: 'HIGH' },
      { type: 'MISCONDUCT', inputStrength: 'HIGH', outputStrength: 'HIGH' },
      { type: 'PROMPT_ATTACK', inputStrength: 'HIGH', outputStrength: 'NONE' },
    ],
  },
});

const agent = new bedrock.CfnAgent(this, 'SupportAgent', {
  agentName: 'customer-support-agent',
  agentResourceRoleArn: agentRole.roleArn,
  foundationModel: 'anthropic.claude-3-haiku-20240307-v1:0',
  instruction: 'You are a customer support agent. Help users with their orders.',
  idleSessionTtlInSeconds: 600,
  guardrailConfiguration: {
    guardrailIdentifier: guardrail.attrGuardrailId,
    guardrailVersion: 'DRAFT',
  },
});
```

### 4.3 Custom Resources for AI Provisioning

Some AI provisioning tasks are not natively supported by CloudFormation (e.g., triggering
a knowledge base sync, running a custom model evaluation). Use CloudFormation Custom
Resources backed by Lambda to fill these gaps.

```typescript
import * as cr from 'aws-cdk-lib/custom-resources';

// Custom resource to trigger knowledge base sync after data source creation
const kbSync = new cr.AwsCustomResource(this, 'KBSync', {
  onCreate: {
    service: 'BedrockAgent',
    action: 'startIngestionJob',
    parameters: {
      knowledgeBaseId: knowledgeBase.attrKnowledgeBaseId,
      dataSourceId: dataSource.attrDataSourceId,
    },
    physicalResourceId: cr.PhysicalResourceId.of('kb-sync-trigger'),
  },
  policy: cr.AwsCustomResourcePolicy.fromSdkCalls({
    resources: cr.AwsCustomResourcePolicy.ANY_RESOURCE,
  }),
});
```

**Exam Tip:** `cdk synth` + `cfn-lint` is the standard validation step in CI/CD for
CDK-based GenAI projects. The exam may ask about validating IaC before deployment -- the
answer is always synthesize first, then validate the generated template.

## 5. AWS CodeArtifact

### 5.1 Package Management for AI Dependencies

CodeArtifact is a fully managed artifact repository service that stores and shares
software packages.

**Key Concepts:**

| Concept | Description |
|---------|-------------|
| **Domain** | Top-level organizational unit; all repositories in a domain share a common policy |
| **Repository** | Stores packages; can have upstream repositories and external connections |
| **External Connection** | Link to a public repository (npmjs.com, PyPI, Maven Central, etc.) |
| **Upstream Repository** | Another CodeArtifact repository that this one can pull from |

**Supported Package Formats:** npm, PyPI, Maven, NuGet, Ruby (gems), Swift, Cargo, Generic

**Architecture Pattern for AI Teams:**

```
External Connections           CodeArtifact Domain
  npmjs.com  ─────> npm-store (repo)
  pypi.org   ─────> pypi-store (repo)
                        |
                    shared-ai-packages (upstream repo)
                        |
                ┌───────┼───────┐
            team-a-repo  team-b-repo  team-c-repo
```

**Why CodeArtifact Matters for GenAI:**
- **Dependency control**: Pin and audit AI/ML library versions (LangChain, boto3, transformers)
- **Security**: Scan packages before they enter your repositories
- **Availability**: Cached packages remain available even if public registries go down
- **Cross-team sharing**: Publish internal AI utility packages (prompt templates, custom
  evaluators) to a shared repository
- **One external connection per repo**: Each repository can connect to only one external
  public repository; use upstream repos to aggregate

**Authentication:**

```bash
# Authenticate npm with CodeArtifact
aws codeartifact login --tool npm \
  --repository my-ai-repo \
  --domain my-domain \
  --domain-owner 123456789012

# Authenticate pip with CodeArtifact
aws codeartifact login --tool pip \
  --repository my-ai-repo \
  --domain my-domain \
  --domain-owner 123456789012
```

**Exam Tip:** CodeArtifact is the answer when the exam asks about managing private AI/ML
package dependencies in an enterprise environment, especially when teams need to share
internal packages while controlling access to public registries.

### 6.2 Scripting AI Workflows

### 7.1 AWS SDK v3 Architecture (Modular Clients)

AWS SDK for JavaScript v3 is the standard for TypeScript/Node.js GenAI development.

**Key Architectural Differences from v2:**

| Feature | SDK v2 | SDK v3 |
|---------|--------|--------|
| Package structure | Monolithic `aws-sdk` | Modular per-service `@aws-sdk/client-*` |

### 7.3 Retry Strategies and Exponential Backoff (Task 2.4.3)

AWS SDKs implement automatic retry with exponential backoff and jitter for transient
errors (throttling, service unavailable, timeouts).

**How Exponential Backoff Works:**

```
Attempt 1: wait 0ms (immediate)
Attempt 2: wait base * 2^1 + jitter  (e.g., ~200ms)
Attempt 3: wait base * 2^2 + jitter  (e.g., ~400ms)
Attempt 4: wait base * 2^3 + jitter  (e.g., ~800ms)
... up to maxAttempts or maxDelay
```

**Jitter** adds randomness to prevent thundering herd: all clients retrying at the same
time after a shared failure.

**SDK v3 Retry Configuration:**

```typescript
import { BedrockRuntimeClient } from '@aws-sdk/client-bedrock-runtime';

const client = new BedrockRuntimeClient({
  region: 'us-east-1',
  // Standard retry mode (default)
  maxAttempts: 3,
});

// Or use adaptive retry mode for high-throughput
const adaptiveClient = new BedrockRuntimeClient({
  region: 'us-east-1',
  retryMode: 'adaptive',  // Adjusts retry rate based on throttling signals
  maxAttempts: 5,
});
```

**Three Retry Modes in SDK v3:**

| Mode | Behavior | When to Use |
|------|----------|-------------|
| `standard` | Exponential backoff with jitter, max 3 attempts | Default; most use cases |
| `adaptive` | Token bucket rate limiting + exponential backoff | High-throughput APIs with throttling |
| `legacy` | Basic retry without jitter | Backward compatibility only |

**Retryable Errors:**
- `ThrottlingException` / HTTP 429 -- Always retry with backoff
- `ServiceUnavailableException` / HTTP 503 -- Retry with backoff
- `InternalServerError` / HTTP 500 -- Retry with backoff
- Network timeouts -- Retry with backoff
- `ModelNotReadyException` -- Retry (model is warming up)

**Non-Retryable Errors:**
- `ValidationException` / HTTP 400 -- Fix the request
- `AccessDeniedException` / HTTP 403 -- Fix IAM permissions
- `ResourceNotFoundException` / HTTP 404 -- Fix the resource ID

**Exam Gotcha:** The exam will test your knowledge of when to use exponential backoff vs.
when to fix the request. Retryable = transient server errors and throttling. Non-retryable
= client errors (4xx except 429).

**Exam Gotcha:** Adaptive retry mode is appropriate when your application makes many
concurrent Bedrock API calls and needs to automatically adjust its request rate to avoid
throttling. Standard mode is sufficient for most single-request patterns.

### 7.4 Middleware Stack

The SDK v3 middleware stack lets you intercept and modify requests/responses at various
stages of the request lifecycle.

**Middleware Stages (in order):**

```
initialize -> serialize -> build -> finalizeRequest -> deserialize
```

| Stage | Purpose | Example Use |
|-------|---------|-------------|
| `initialize` | Set defaults, validate input | Add default inference config |
| `serialize` | Convert input to HTTP request | Custom serialization |
| `build` | Add headers, sign request | Add custom tracing headers |
| `finalizeRequest` | Last chance before sending | Logging, metrics emission |
| `deserialize` | Parse HTTP response | Custom response processing |

**Custom Middleware Example -- Logging All Bedrock Calls:**

```typescript
import { BedrockRuntimeClient, ConverseCommand } from '@aws-sdk/client-bedrock-runtime';

const client = new BedrockRuntimeClient({ region: 'us-east-1' });

// Add middleware to log all requests and responses
client.middlewareStack.add(
  (next, context) => async (args) => {
    const startTime = Date.now();
    console.log('Request:', {
      commandName: context.commandName,
      input: args.input,
    });

    const result = await next(args);

    console.log('Response:', {
      commandName: context.commandName,
      duration: Date.now() - startTime,
      metadata: result.output.$metadata,
    });

    return result;
  },
  {
    step: 'initialize',
    name: 'bedrockLogger',
    tags: ['LOGGING'],
  }
);
```

**Middleware for Token Usage Tracking:**

```typescript
client.middlewareStack.add(
  (next, context) => async (args) => {
    const result = await next(args);

    if (context.commandName === 'ConverseCommand') {
      const usage = result.output.usage;
      if (usage) {
        // Emit CloudWatch custom metric
        console.log(JSON.stringify({
          _aws: { Timestamp: Date.now(), CloudWatchMetrics: [{
            Namespace: 'GenAI/TokenUsage',
            Dimensions: [['ModelId']],
            Metrics: [
              { Name: 'InputTokens', Unit: 'Count' },
              { Name: 'OutputTokens', Unit: 'Count' },
            ],
          }]},
          ModelId: args.input.modelId,
          InputTokens: usage.inputTokens,
          OutputTokens: usage.outputTokens,
        }));
      }
    }

    return result;
  },
  { step: 'deserialize', name: 'tokenTracker' }
);
```

**Exam Tip:** The middleware stack is how you add cross-cutting concerns (logging, metrics,
tracing, retry customization) without modifying business logic. The exam may describe a
scenario where you need to add observability to all Bedrock calls -- middleware is the
answer.

## Exam Gotchas Summary

1. **X-Ray annotations vs. metadata**: Annotations are indexed and searchable (max 50 per trace). Metadata is not indexed but can store any JSON object. For AI traces, put `modelId` and `useCase` in annotations; put prompts and responses in metadata.

2. **X-Ray in Lambda**: Lambda creates the root segment. Your code creates subsegments only. You cannot create new root segments in Lambda.

3. **X-Ray SDK -> OpenTelemetry migration**: X-Ray SDK enters maintenance mode February 2026. ADOT (AWS Distro for OpenTelemetry) is the recommended replacement. Bedrock AgentCore uses ADOT natively.

4. **Amplify AI Kit route types**: `a.conversation()` = multi-turn, async, streaming, DynamoDB-backed. `a.generation()` = single-turn, sync, no storage. Both use Bedrock's Converse API.

5. **Amplify AI Kit authorization**: Conversation routes use owner-based auth (each user sees only their conversations). This is architectural, not a configuration choice.

6. **CI/CD for prompts**: Prompts are versioned assets in source control. They go through regression testing (golden test cases) before deployment. CodePipeline approval gates are used for human review of prompt changes.

7. **CodeDeploy canary for Lambda**: Requires `AutoPublishAlias`. First deploy creates the alias; second deploy onward uses gradual traffic shifting. Uses weighted aliases, not API Gateway canaries.

8. **CloudFormation Bedrock resources**: `AWS::Bedrock::Agent`, `AWS::Bedrock::KnowledgeBase`, `AWS::Bedrock::Guardrail`, `AWS::Bedrock::DataSource`, `AWS::Bedrock::Flow` are all available as native CFN resources.

9. **SDK v3 retry modes**: `standard` (default, 3 attempts), `adaptive` (token bucket for high throughput), `legacy` (backward compat). Exponential backoff with jitter prevents thundering herd.

10. **SDK v3 retryable vs. non-retryable**: Retry 429 (throttle), 500, 503. Do NOT retry 400 (validation), 403 (access denied), 404 (not found).

11. **CodeArtifact**: One external connection per repository. Use upstream repos to aggregate multiple public sources. This is the answer for enterprise AI/ML dependency management.

12. **`cdk synth` + `cfn-lint`**: Standard IaC validation step in CI/CD. Always validate synthesized templates before deployment.
