# Project 10: CI/CD Pipeline for GenAI Application

## Exam Domains

| Domain | Task | Description |
|--------|------|-------------|
| 2.3 | Design and implement enterprise integration architectures | CI/CD pipelines, CodePipeline, CodeBuild, automated testing frameworks, security scans, rollback support |
| 5.1 | Implement evaluation systems for GenAI | Deployment validation systems, synthetic user workflows, AI-specific output validation, automated quality gates |

## Overview

Deploying a GenAI application without a structured release pipeline means every change is a manual, unverified risk — a prompt wording change can silently degrade answer quality, and a misconfigured guardrail can expose the model to inputs it should block. The AIP-C01 exam tests CI/CD specifically for GenAI workloads, which have unique quality gates that traditional pipelines do not address: prompt regression testing (did the model's answers change semantically?), guardrail validation, and automated rollback when output quality degrades.

This project defines the GenAI application infrastructure as code, then builds a CodePipeline that runs those quality gates automatically on every commit. The pipeline includes a prompt regression test stage in CodeBuild that embeds baseline and current responses and rejects the deployment if cosine similarity drops below a threshold — a concrete implementation of the "deployment validation" pattern described in Task 5.1.9.

## Architecture

```
GitHub (or CodeCommit) repo
    |  push to main
    v
CodePipeline
    |
    |-- Stage 1: Source
    |       CodeStar Connections (GitHub) or CodeCommit trigger
    |
    |-- Stage 2: Build
    |       CodeBuild: validate infrastructure templates
    |           --> validates infrastructure templates
    |           --> runs unit tests for Lambda functions
    |           --> outputs CloudFormation templates to artifact store
    |
    |-- Stage 3: PromptRegression
    |       CodeBuild: prompt_regression.py
    |           --> for each prompt in regression_suite.json:
    |               - invoke current Lambda (pre-deploy)
    |               - embed baseline response (stored in S3 baseline/)
    |               - embed current response
    |               - compute cosine similarity
    |               - fail build if similarity < 0.85 for any prompt
    |           --> publish test results to CodeBuild test reports
    |
    |-- Stage 4: GuardrailValidation
    |       CodeBuild: validate_guardrails.py
    |           --> for each test case in guardrail_test_suite.json:
    |               - invoke Bedrock Guardrail via ApplyGuardrail API
    |               - assert expected action (BLOCKED or NONE)
    |           --> fail build if any assertion fails
    |
    |-- Stage 5: Deploy
    |       CloudFormation ChangeSet (from infrastructure templates)
    |           --> deploys Lambda, API Gateway, Bedrock config updates
    |
    |-- Stage 6: PostDeployValidation
            CodeBuild: post_deploy_smoke.py
                --> invoke deployed API with 3 smoke test questions
                --> assert non-empty responses and HTTP 200
                --> on failure: trigger CloudFormation rollback via API

GenAI Application Stack:
    Lambda (chat handler) + API Gateway + Bedrock Knowledge Base config + Guardrail
    S3 (baseline responses)
    CodePipeline + all stages
```

**AWS services used:**

- Infrastructure as Code tool of your choice -- defines the GenAI application stack
- AWS CodePipeline -- orchestrates the multi-stage release pipeline
- AWS CodeBuild -- runs build, test, and validation stages
- AWS CodeCommit or GitHub (via CodeStar Connections) -- source repository
- AWS Lambda -- the GenAI application being deployed
- Amazon Bedrock -- model invocation and guardrail validation
- Amazon S3 -- artifact store, baseline response storage
- Amazon CloudFormation -- deployment target for infrastructure templates
- Amazon CloudWatch -- CodeBuild test reports, pipeline metrics
- AWS IAM -- CodePipeline and CodeBuild execution roles
- Amazon SNS -- optional: pipeline failure notifications

## What You Will Build

1. A GenAI application stack containing: a Lambda chat handler, an API Gateway HTTP API, and Bedrock resource references (knowledge base ID, guardrail ID stored as SSM parameters). The infrastructure is defined as code and produces deployable templates.
2. A `regression_suite.json` file with 5 prompt-baseline pairs. Baseline responses are pre-generated by running the application and saved to S3 during an initial bootstrap step.
3. A `prompt_regression.py` CodeBuild script that loads the regression suite, invokes the Lambda (using its function URL or via API Gateway), embeds both the baseline and current response with Titan Text Embeddings v2, computes cosine similarity, and writes a JUnit XML report (for CodeBuild test report integration).
4. A `guardrail_test_suite.json` with 6 test cases (3 should-block, 3 should-pass) and a `validate_guardrails.py` CodeBuild script that calls `bedrock:ApplyGuardrail` and asserts the expected `action` field.
5. A `post_deploy_smoke.py` script that makes 3 HTTP requests to the deployed API Gateway URL and asserts non-empty, non-error responses. On any failure, calls the CloudFormation API to cancel the update and roll back.
6. A complete CodePipeline with all stages wired together, including a manual approval action between the PromptRegression stage and the Deploy stage (optional but recommended for exam pattern reinforcement).
7. A CodeBuild test report group that receives the JUnit XML from `prompt_regression.py` and displays pass/fail results in the CodeBuild console.
8. A CloudWatch dashboard `GenAI-Pipeline` showing: pipeline execution success/failure rate, regression test pass rate over time (from CodeBuild metrics), and deployment frequency.

## Target Outcomes

By the end of this project you should be able to verify each of the following:

- [ ] Pushing a commit that changes only a non-prompt code path (e.g., logging format) completes the regression stage with all 5 prompts passing similarity checks, and the pipeline proceeds to deploy.
- [ ] Pushing a commit that deliberately degrades a prompt (e.g., removing the system prompt that enforces answer format) fails the regression stage — the CodeBuild build shows a FAILED status and the pipeline stops before deploying.
- [ ] The guardrail validation stage correctly asserts that blocked test cases return `GUARDRAIL_INTERVENED` and non-blocked cases return `NONE`.
- [ ] After a successful deploy, the post-deploy smoke test passes and the API Gateway URL returns responses to all 3 smoke test questions.
- [ ] The CodeBuild test report shows individual pass/fail status for each regression prompt and each guardrail test case.
- [ ] You can explain why semantic similarity (cosine similarity of embeddings) is a better regression metric for LLM outputs than exact string comparison.

## Step-by-Step Build Guide

1. **Define the GenAI application stack.** Using your preferred IaC tool, define: a Lambda function (`chat-handler`) that accepts `{ prompt }` and calls `bedrock:InvokeModel` with Claude 3 Haiku, an HTTP API Gateway with a `POST /chat` route, and two SSM `StringParameter` entries for `KnowledgeBaseId` and `GuardrailId` (placeholder values to start). The stack should produce deployable infrastructure templates.

2. **Bootstrap the baseline.** Deploy the GenAI application stack. Send 5 prompts to the API Gateway endpoint. Save each response to S3 at `s3://<bucket>/baselines/<prompt_id>.json`. Write `tests/regression_suite.json` with format `[{ id, prompt, baseline_s3_key }]`.

3. **Write `tests/prompt_regression.py`.** For each entry in the regression suite: invoke the API Gateway endpoint (or Lambda function URL), load baseline text from S3, embed both responses with Titan Text Embeddings v2 via `bedrock:InvokeModel`, compute cosine similarity using numpy. If similarity < 0.85, mark FAILED. Write a JUnit XML report to `test-results/regression-results.xml`. Exit with code 1 if any test failed.

4. **Write `tests/validate_guardrails.py`.** Load `guardrail_test_suite.json` (6 test cases: 3 should-block, 3 should-pass). For each, call `bedrock:ApplyGuardrail`. Assert `action == expected_action`. Write JUnit XML. Exit 1 on any failure.

5. **Write `tests/post_deploy_smoke.py`.** Accept `API_URL` env var. Send 3 HTTP requests. Assert HTTP 200 and non-empty `response` field. On failure, call `cloudformation:CancelUpdateStack` with the stack name to trigger rollback.

6. **Create the CodePipeline** with the following stages:
   - **Stage 1 (Source):** CodeStar Connections source action (GitHub) or CodeCommit trigger.
   - **Stage 2 (Build):** CodeBuild project running `buildspec.yml` -- validates infrastructure templates, runs unit tests for Lambda functions, outputs deployment artifacts.
   - **Stage 3 (PromptRegression):** CodeBuild project running `buildspec-regression.yml` -- installs Python deps, runs `prompt_regression.py`, uploads JUnit XML to a CodeBuild Report Group.
   - **Stage 4 (GuardrailValidation):** CodeBuild project running `buildspec-guardrail.yml` -- runs `validate_guardrails.py`.
   - **Stage 5 (ManualApproval):** Manual approval action (optional but recommended for exam pattern reinforcement).
   - **Stage 6 (Deploy):** CloudFormation CreateChangeSet + ExecuteChangeSet actions using the templates from the Build stage.
   - **Stage 7 (PostDeployValidation):** CodeBuild project running `buildspec-smoke.yml` -- runs `post_deploy_smoke.py`.
   - **IAM roles:** regression stage needs `lambda:InvokeFunction`, `bedrock:InvokeModel`, `s3:GetObject`; guardrail stage needs `bedrock:ApplyGuardrail`; smoke stage needs `execute-api:Invoke` and `cloudformation:CancelUpdateStack`.

7. **Create a CodeBuild Report Group** to receive the JUnit XML from `prompt_regression.py` and `validate_guardrails.py`.

8. **Deploy all pipeline resources to your AWS account** using your preferred infrastructure-as-code tool or the AWS Management Console.

9. **Commit and push** to trigger the pipeline. Observe execution. Fix any IAM or buildspec issues.

10. **Force a regression failure** by modifying a prompt in the chat handler (e.g., remove the system prompt that enforces answer format). Push and verify the PromptRegression stage fails and the pipeline stops before the Deploy stage.

11. **Build a CloudWatch dashboard** named `GenAI-Pipeline` using CodePipeline execution metrics and CodeBuild test report metrics. Widgets: pipeline execution success/failure rate, regression test pass rate over time, and deployment frequency.
