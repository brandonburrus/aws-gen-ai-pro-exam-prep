# Domain 4: Operational Efficiency and Optimization for GenAI Applications

**Exam Weight: 12%**

This domain covers three task areas: cost optimization and resource efficiency, application
performance optimization, and monitoring systems for GenAI applications. Despite being the
lowest-weighted domain, it is dense with testable AWS-specific details around Bedrock pricing
mechanics, caching features, scaling patterns, and observability tooling.

## Task 4.1 -- Implement Cost Optimization and Resource Efficiency Strategies

### Key Concepts

**Token Efficiency Systems (Skill 4.1.1)**

Amazon Bedrock charges per token (input and output). Every unnecessary token directly increases
cost, making token efficiency the single most impactful lever for cost control.

**Token Tracking and Monitoring**

Bedrock publishes token metrics to CloudWatch under the `AWS/Bedrock` namespace:

| Metric | Description |
|---|---|
| `InputTokenCount` | Tokens provided as input to the model |
| `OutputTokenCount` | Tokens generated by the model |
| `CacheReadInputTokens` | Tokens retrieved from prompt cache (not reprocessed) |
| `CacheWriteInputTokens` | Tokens written into the prompt cache |
| `EstimatedTPMQuotaUsage` | Quota consumption including cache write tokens and burndown |

Pre-configured CloudWatch dashboards show invocation count, token counts by model, daily token
trends, input vs. output distributions, and requests grouped by input token count.

**Burndown Rate**

For Anthropic Claude models v3.7 and later, output tokens have a **5x burndown rate** -- meaning
1 output token consumes 5 tokens from the TPM/TPD quota. All other models have a 1:1 rate.

Example: A request with 1,000 input tokens and 100 output tokens on Claude Sonnet 4 depletes
**1,500 tokens** from quota (1,000 + 100 x 5), but you are only **billed** for 1,100 tokens.
The quota deduction and billing amount differ.

**The `max_tokens` Parameter**

This parameter is deducted from quota at the **start** of each request, before any output is
generated. Setting it too high (e.g., 32,000 when actual output averages 500) artificially
reserves quota capacity, reducing your available concurrency. Unused tokens are replenished
when the request completes, but during execution those reserved tokens count against your quota.

Optimization: use CloudWatch `OutputTokenCount` metrics to study actual output sizes, then set
`max_tokens` to a value slightly above the observed maximum.

> **Exam tip:** If a question describes throttling despite low actual token usage, the answer
> likely involves `max_tokens` being set too high or the burndown rate consuming quota faster
> than expected.

**Context Window Optimization**

Each model has a finite context window (e.g., Claude 3.5 Sonnet: 200K tokens, Amazon Nova Pro:
300K tokens). Techniques to optimize context usage:

- Send only relevant context: for RAG, retrieve only the top-k most relevant chunks rather than
  stuffing the full knowledge base.
- Manage conversation history: implement sliding-window approaches that keep only the most
  recent N turns, or summarize older turns before including them.
- Use structured output instructions: tell the model exactly what format you need, reducing
  unnecessary back-and-forth.

**Prompt Compression and Context Pruning**

Ranked by impact:

1. **Amazon Bedrock Prompt Optimization** -- a managed feature that automatically optimizes
   prompts for a target LLM and task. Integrated into Bedrock Playground and Prompt Management.
2. **Manual prompt engineering** -- remove filler words, redundant instructions, and unnecessary
   context. The Well-Architected GenAI Lens (GENCOST03-BP01) recommends making prompts as short
   as possible while meeting performance requirements.
3. **LLM-based compression** -- use a cheaper model to summarize or compress a prompt before
   sending it to the primary model. Trades a small additional inference cost for significant
   savings on the main call.
4. **Prompting technique selection** -- zero-shot eliminates the token cost of examples entirely.
   Chain-of-thought and tree-of-thought add output tokens but improve reasoning. Choose the
   cheapest technique that meets accuracy requirements.
5. **Context pruning in RAG** -- limit retrieved chunks, use reranking to select only the most
   relevant results, strip metadata and formatting from documents before injection.

**Response Size Controls**

- Set `max_tokens` to the minimum needed for the use case.
- Use explicit instructions: "respond in 3 sentences" or "return only JSON."
- Provide an output schema that constrains response length.
- For streaming, implement client-side truncation when sufficient information is received.

### Cost-Effective Model Selection Frameworks (Skill 4.1.2)

The fundamental principle: not every request needs the most capable (and expensive) model.

**Amazon Bedrock Intelligent Prompt Routing**

Provides a single serverless endpoint that dynamically routes each request to the most suitable
model within a model family, predicting response quality per request.

| Aspect | Detail |
|---|---|
| Supported families | Anthropic Claude (Haiku + Sonnet), Meta Llama (8B + 70B), Amazon Nova (Lite + Pro) |
| Cost savings | Up to 30% without compromising accuracy (internal testing showed 48% for Claude) |
| Router types | Default (pre-configured) and Configured (user-specified criteria and fallback model) |
| Limitations | English only, pairwise routing (two models), cannot adapt to app-specific metrics |
| Pricing | No charge for routing itself; pay on-demand price for whichever model processes the request |

**Amazon Bedrock On-Demand Service Tiers**

Four tiers map workload priority to cost:

| Tier | Use Case | Key Characteristic |
|---|---|---|
| Priority | Mission-critical, customer-facing | Up to 25% better OTPS latency vs. Standard |
| Standard | General AI applications | Consistent performance baseline |
| Flex | Non-urgent: evaluations, batch-like workloads | Discounted rate, higher throttling risk |
| Reserved | Predictable, consistent throughput | 1-month or 3-month commitments, guaranteed capacity |

You select the tier on a per-API-call basis, allowing workload-level optimization.

**Amazon Bedrock Model Distillation**

Transfers knowledge from a larger "teacher" model to a smaller "student" model:

- Distilled models are up to **500% faster** and up to **75% less expensive** with less than 2%
  accuracy loss for use cases like RAG.
- Supported flows include Amazon (Nova Premier to Nova Pro), Anthropic (Sonnet to Haiku), and
  Meta (70B to 1B/3B).
- Input: JSONL file in S3 or historical invocation logs.
- Deployment: distilled model requires Provisioned Throughput for inference.

> **Exam tip:** Model Distillation is the answer when a question asks "how to get a cheaper
> model with similar accuracy for my specific use case." Intelligent Prompt Routing is the
> answer when the question asks "how to route different requests to the right model automatically."

**SageMaker Inference Cost Optimization**

For self-hosted models:

- **Inference Recommender** benchmarks your model across instance types for optimal
  price-performance.
- **SageMaker Savings Plans** offer up to 64% cost reduction for consistent usage.
- **Multi-model endpoints (inference components)** deploy multiple models on a single endpoint
  to improve GPU utilization.
- **Serverless inference** provides pay-per-invocation pricing for intermittent traffic.

### High-Performance FM Systems for Throughput (Skill 4.1.3)

**Bedrock Batch Inference**

For large-scale, non-real-time workloads:

- Submit JSONL input files to S3 (up to 10,000 records, 200 MB max). Create a batch job.
  Results are written to an output S3 bucket asynchronously within a 24-hour window.
- **Up to 50% less** than on-demand pricing.
- Batch inference is **not supported for Provisioned Throughput** models.
- Monitoring: CloudWatch metrics in `AWS/Bedrock/Batch` namespace, EventBridge notifications
  for job state changes.

**SageMaker Inference Options**

| Option | Best For | Payload Limit | Timeout | Scale to Zero |
|---|---|---|---|---|
| Real-time | Low latency, predictable traffic | 25 MB | 60s (8 min streaming) | No |
| Serverless | Intermittent/unpredictable traffic | 4 MB | 60s | Yes (automatic) |
| Asynchronous | Large payloads, long processing | 1 GB | 1 hour | Yes |
| Batch Transform | Offline processing, large datasets | No limit | Days | N/A (no endpoint) |

**Bedrock Provisioned Throughput**

- Measured in **Model Units (MUs)**, each delivering a specific token throughput per minute.
- Commitment options: no commitment, 1-month, or 6-month (longer = cheaper).
- Use for high, predictable, consistent workloads that need guaranteed performance.
- Must request MU quota through AWS Support before purchasing.
- Specify the provisioned model ARN as `modelId` in API calls.

**Cross-Region Inference**

Routes requests across multiple regions within a geography for higher quotas:

- Geographic profiles (e.g., `us.`) route within a boundary, standard pricing.
- Global profiles (e.g., `global.`) route anywhere worldwide, ~10% cost savings, highest throughput.
- No additional cost for routing itself.

**Auto-Scaling for SageMaker Endpoints**

- **Target-tracking**: use `SageMakerVariantInvocationsPerInstance` as the predefined metric.
- **Step scaling**: define thresholds and scale amounts.
- **Scheduled scaling**: pre-scale for known traffic patterns.
- **Scale to zero**: supported for async endpoints using the `HasBacklogWithoutCapacity`
  CloudWatch alarm.
- Configure cooldown periods to prevent thrashing.

### Intelligent Caching Systems (Skill 4.1.4)

There are two distinct caching layers, and the exam tests both.

**Layer 1: Amazon Bedrock Prompt Caching (Model-Level)**

Caches the internal model state for prompt prefixes, avoiding recomputation on subsequent calls.

How it works:
1. Mark portions of a prompt as **cache checkpoints**.
2. Everything before the checkpoint becomes the cached prompt prefix.
3. Subsequent requests with the same prefix skip recomputation.
4. The prefix must be **static** -- any change causes a cache miss.

| Aspect | Detail |
|---|---|
| Cost reduction | Up to **90%** on input token costs |
| Latency improvement | Up to **85%** reduction |
| TTL | 5-minute default (resets on each hit); some models support 1-hour extended TTL |
| Supported APIs | Converse, ConverseStream, InvokeModel, InvokeModelWithResponseStream |
| Supported models | Claude (Opus 4.5 through 3.5 Haiku), Amazon Nova (Micro through Premier) |

Pricing mechanics:
- **Cache writes** are charged at a higher rate than standard input tokens.
- **Cache reads** are charged at a **reduced rate** (up to 90% less).
- Net savings depend on your cache hit rate.

Best use cases: chat with document (cache the document, vary the question), few-shot learning
(cache system prompt + examples), coding assistants (cache code context), agentic workflows
(cache tool definitions).

**Layer 2: Semantic Caching (Application-Level)**

Stores complete request-response pairs and **bypasses the LLM entirely** for matching queries.

**Semantic Caching with ElastiCache for Valkey:**
1. Convert the incoming query into a vector embedding.
2. Search the cache for semantically similar previous queries using vector similarity.
3. If a match exceeds the similarity threshold, return the cached response (no LLM call).
4. If no match, invoke the LLM, store the query embedding + response in the cache.

ElastiCache for Valkey is the AWS-recommended store due to lowest latency vector search with
95%+ recall. Supports HNSW and FLAT search algorithms, cosine/Euclidean/inner product distance.

Alternative stores:
- **Amazon MemoryDB for Redis** -- durable semantic caching (persists across restarts).
- **Amazon DynamoDB** -- persistent cache with flexible querying.
- **Amazon OpenSearch Service** -- hybrid full-text and vector search.

**Deterministic Request Hashing**

Hash the exact request (prompt + parameters) as a cache key. Look up in ElastiCache or
DynamoDB. Suitable when queries repeat verbatim (automated pipelines, templated prompts).
Simpler than semantic caching but misses paraphrased queries.

**Edge Caching with CloudFront**

Cache deterministic API responses at edge locations. Use CloudFront with API Gateway as origin.
Appropriate for static/semi-static queries (product descriptions, FAQ responses). NOT suitable
for personalized or conversational responses.

> **Exam tip:** Bedrock prompt caching and semantic caching solve different problems. Prompt
> caching caches the model's internal computation state for repeated prefixes. Semantic caching
> stores full responses and skips the model entirely. If the question says "reduce LLM calls for
> similar queries," think semantic caching. If it says "reduce latency for repeated prompt
> context," think prompt caching.

### Cost Optimization Decision Framework

| Problem | Solution |
|---|---|
| Prompts are too long/verbose | Prompt compression, Bedrock Prompt Optimization, context pruning |
| Paying for capability you don't need | Intelligent Prompt Routing, tiered model selection, Distillation |
| High volume, non-urgent workloads | Batch inference (50% savings), Flex tier |
| Same prompt prefixes processed repeatedly | Bedrock prompt caching (90% input cost reduction) |
| Same/similar questions asked by users | Semantic caching with ElastiCache (bypass LLM entirely) |
| Unpredictable traffic causing over-provisioning | On-demand pricing, SageMaker auto-scaling, scale-to-zero |
| High predictable traffic | Provisioned Throughput, Reserved tier, SageMaker Savings Plans |
| Output tokens consuming too much quota | Reduce `max_tokens`, use response size controls |
| Multi-model deployment wasting GPUs | SageMaker inference components (multi-model endpoints) |
| Need cheaper model for specific use case | Model Distillation (teacher to student) |

## Task 4.2 -- Optimize Application Performance

### Key Concepts

**Responsive AI Systems and Latency Optimization (Skill 4.2.1)**

Three primary latency metrics for LLM applications:

- **Time to First Token (TTFT)**: latency from when Bedrock receives a streaming request to
  the first response token. Most critical for perceived responsiveness.
- **Output Tokens Per Second (OTPS)**: rate at which tokens are generated after the first.
  Determines how quickly the full response appears.
- **End-to-End Latency (E2E)**: total time from request submission to final token receipt.

**Latency-Optimized Inference in Bedrock**

Configured via the `performanceConfig` parameter:

```json
"performanceConfig": {
    "latency": "optimized"
}
```

- Requires no additional setup or fine-tuning.
- Uses purpose-built AI chips (AWS Trainium2) and software optimizations.
- If latency-optimized quota is exceeded, requests **fall back to standard latency** (no error).
- CloudWatch metrics tracked under `model-id+latency-optimized`.

**Response Streaming**

Two streaming APIs:
- `InvokeModelWithResponseStream` (model-specific format)
- `ConverseStream` (unified API -- recommended)

Streaming is critical for perceived performance. Users see tokens as they are generated. The
`ConverseStream` response returns events: `messageStart`, `contentBlockStart`,
`contentBlockDelta`, `contentBlockStop`, `messageStop`, `metadata`.

Streaming does **not** change the total cost -- you pay the same tokens regardless. It only
improves perceived latency.

**Pre-computation Techniques**

- **Prompt caching**: cache frequently used prefixes (up to 85% latency reduction).
- **Semantic caching**: cache full responses for similar queries (bypasses LLM entirely).
- **Edge caching**: serve deterministic responses closer to users via CloudFront.

**Parallel Requests for Complex Workflows**

- Use **Step Functions** parallel states for independent sub-tasks.
- Bedrock Agents can invoke tools in parallel where tasks are independent.
- Map-reduce: split a document into chunks, process in parallel, combine results.

### Retrieval Performance Enhancement (Skill 4.2.2)

Retrieval performance depends on three factors: **index quality**, **query quality**, and
**ranking quality**.

**Index Optimization for Vector Stores**

*Amazon OpenSearch Service:*
- Vector engines: FAISS, NMSLIB, Lucene.
- Two search methods: **exact k-NN** (brute force) and **approximate k-NN** (HNSW -- faster
  for large datasets, recommended for production).
- **Auto-optimize**: specify acceptable latency and recall thresholds; OpenSearch evaluates
  and recommends optimal configurations. Results in under an hour.
- **GPU-accelerated indexing**: up to 10x faster index builds at 1/4 the cost.
- **Disk-based vector search**: keeps reduced-precision vectors in memory for candidate
  generation, retrieves full-precision vectors for scoring. Runs at 1/3 the cost.
- Distance metrics: Euclidean, cosine similarity, dot product.

*Amazon Aurora PostgreSQL (pgvector):*
- **HNSW** index: better query performance, higher memory, slower build. Preferred for
  production.
- **IVFFlat** index: faster build, lower memory, slightly lower recall.
- **Iterative index scans** (pgvector 0.8.0): solve the "overfiltering" problem where
  combining vector similarity with WHERE clauses returns too few results. Use `relaxed_order`
  mode for best balance.

**Query Preprocessing Techniques**

- **Query decomposition**: Bedrock Knowledge Bases break complex queries into sub-queries,
  retrieve independently, then combine results.
- **Implicit metadata filtering**: Bedrock auto-generates filters from the user query and a
  metadata schema.
- **Manual metadata filtering**: explicit filters (equals, not equals, greater than, less than,
  in) to narrow the search space before vector comparison.

**Hybrid Search**

Bedrock Knowledge Bases search types:
- `Default`: Bedrock decides the strategy.
- `Hybrid`: combines vector embeddings (semantic) with raw text (lexical/keyword). **Only
  supported for Amazon RDS, OpenSearch Serverless, and MongoDB with a filterable text field.**
- `Semantic`: pure vector search.

**Reranking**: a second-pass that uses an LLM to reorder initial vector search results.
Configure via `VectorSearchBedrockRerankingConfiguration`. Set `numberOfRerankedResults` (1-100)
-- fewer results = faster but may exclude relevant content.

> **Exam tip:** Hybrid search requires a filterable text field. Without one, Bedrock falls back
> to semantic-only. If a question mentions "keyword search not working in Knowledge Bases,"
> check whether the vector store has a filterable text field configured.

### FM Throughput Optimization (Skill 4.2.3)

Throughput is measured in:
- **Requests Per Minute (RPM)**: individual API calls allowed.
- **Tokens Per Minute (TPM)**: total tokens (input + output) per minute.
- **Tokens Per Day (TPD)**: daily token budget (default = TPM x 1,440).

**Handling Throttling (429 ThrottlingException)**

- **Exponential backoff with jitter** on retries.
- **Token-aware rate limiting**: track consumption client-side before sending.
- **Right-size quotas**: request increases through Service Quotas.
- Break large tasks into smaller chunks to avoid token-based throttling.
- Use streaming responses to process output incrementally.

**Capacity Tiers (On-Demand)**

| Tier | RPM Range | TPM Range | Throttling Risk |
|---|---|---|---|
| Flex | 10-100 | 5K-50K | High |
| Standard | 100-500 | 50K-150K | Medium |
| Priority | 500-1000+ | 150K-300K+ | Low |

**Cross-Region Inference for Throughput**

- Geographic profiles: route within a geographic boundary, standard pricing.
- Global profiles: route to any commercial region, ~10% cost savings, highest throughput.
- No additional cost for the routing itself.

### FM Parameter Tuning (Skill 4.2.4)

**Randomness and Diversity Parameters**

| Parameter | Lower Value Effect | Higher Value Effect |
|---|---|---|
| Temperature | More deterministic, higher-probability outputs | More creative/random, lower-probability outputs |
| Top K | Smaller candidate pool, more predictable | Larger candidate pool, more diverse |
| Top P | Only highest-probability tokens | Broader set of tokens considered |

**Parameter Selection Guidelines**

| Use Case | Temperature | Top P | Top K |
|---|---|---|---|
| Factual Q&A, data extraction | 0.0-0.3 | 0.1-0.3 | 1-10 |
| Customer service | 0.3-0.5 | 0.5-0.7 | 20-50 |
| Creative writing, brainstorming | 0.7-1.0 | 0.8-0.95 | 100+ |
| Code generation | 0.1-0.3 | 0.2-0.5 | 5-20 |
| Summarization | 0.3-0.5 | 0.5-0.8 | 20-50 |

**Length and Control Parameters**

- `maxTokens`: maximum tokens in the response. Affects quota reservation (see 4.1.1).
- `stopSequences`: character sequences that halt generation.

**The Converse API** is the recommended unified interface for inference parameters. Use
`inferenceConfig` for standard parameters (`maxTokens`, `temperature`, `topP`, `stopSequences`)
and `additionalModelRequestFields` for model-specific parameters.

**A/B Testing**

- **Bedrock Model Evaluations**: compare model performance with automatic metrics, human
  evaluation, or LLM-as-a-Judge.
- **Application inference profiles**: tag and track costs/performance per configuration.

> **Exam tip:** Parameter tuning does **not change the model** -- it only changes inference
> behavior. Temperature = 0 maximizes determinism but is not guaranteed to be fully deterministic
> for all models. The Converse API is the preferred interface for setting parameters across
> models.

### Resource Allocation for FM Workloads (Skill 4.2.5)

**Capacity Planning**

- Use CloudWatch `EstimatedTPMQuotaUsage` to track actual quota consumption.
- Set alarms at **70-80% utilization** to proactively request increases.
- Account for burndown rate (5x for Claude 3.7+ output) when planning.
- Transition to Reserved when consistent load exceeds ~40% of on-demand costs.

**SageMaker Endpoint Auto-Scaling**

- **Target-tracking**: use `SageMakerVariantInvocationsPerInstance` or custom metrics.
- **Async endpoints**: can scale to zero using `HasBacklogWithoutCapacity` alarm.
- **Serverless endpoints**: auto-scale Provisioned Concurrency using
  `SageMakerVariantProvisionedConcurrencyUtilization`.
- Configure cooldown periods (default 300s) to prevent rapid fluctuation.

**Lambda Concurrency for GenAI**

- Default: 1,000 concurrent executions per Region across all functions.
- **Reserved concurrency**: sets a maximum for a specific function. Use to **limit** FM call
  rate and prevent exceeding Bedrock RPM/TPM quotas.
- **Provisioned concurrency**: pre-initializes execution environments to eliminate cold starts.
  Can auto-scale using Application Auto Scaling.

> **Exam tip:** Reserved concurrency on Lambda is a **rate-limiting mechanism** for downstream
> FM calls. This is a commonly tested pattern. Bedrock handles its own scaling for on-demand
> inference -- you do not configure scaling policies for Bedrock itself.

### System Performance Optimization (Skill 4.2.6)

**API Call Profiling**

- Enable **Bedrock Model Invocation Logging** (disabled by default) to capture full
  request/response data.
- Send logs to CloudWatch Logs (real-time analysis via Logs Insights) and/or S3.
- Use **application inference profiles** to tag and track specific usage patterns by team
  or application.

Key profiling questions:
- What is the input/output token ratio? (Guides `max_tokens` sizing.)
- Which requests are throttled? (Guides tier selection.)
- What is the TTFT distribution? (Guides latency optimization.)
- Are there time-based patterns in token consumption? (Guides scheduled scaling.)

**Vector Database Query Optimization**

- OpenSearch: use HNSW, quantization, disk-based search, auto-optimize.
- Aurora pgvector: use HNSW indexes, enable iterative scans for filtered queries.
- Bedrock Knowledge Bases: use reranking, metadata filtering, query decomposition, hybrid search.

**Latency Reduction Techniques for LLM Inference**

1. Response streaming (`ConverseStream`).
2. Latency-optimized inference (`performanceConfig.latency = "optimized"`).
3. Prompt caching (up to 85% latency reduction).
4. Intelligent Prompt Routing (route simple queries to faster models).
5. Cross-region inference (route to less-busy regions).
6. Smaller context windows and model selection (use Haiku for simple tasks).

**Efficient Service Communication Patterns**

- **Asynchronous**: SQS + Lambda for decoupled, rate-limited FM invocations. Set Lambda
  concurrency to match Bedrock RPM limits.
- **Circuit breaker**: stop sending requests on 503 (ServiceUnavailable) to avoid cascading
  failures. Implement with Step Functions or client-side logic.
- **Exponential backoff with jitter**: standard retry for 429 (Throttling).
- **Fan-out/fan-in**: Step Functions parallel states for independent sub-tasks.
- **Event-driven**: EventBridge + Lambda for async GenAI processing.

**X-Ray Tracing**

- **Segments**: processing data per service. **Subsegments**: downstream calls within a segment.
- **Traces**: end-to-end request path (e.g., API Gateway to Lambda to Bedrock to DynamoDB).
- **Annotations**: indexed key-value pairs for filtering (model_id, token_count).
- **Metadata**: non-indexed data for debugging (prompt content, response content).
- **Adaptive Sampling**: dynamically increases sampling during anomalies. Includes Sampling
  Boost (rate increase during anomalies) and Anomaly Span Capture (captures critical spans
  even without full traces). 1-minute boost window, 10-minute cooldown.

> **Exam tip:** X-Ray annotations are indexed and searchable; metadata is not. The circuit
> breaker pattern is for 503 errors; exponential backoff is for 429 errors. SQS + Lambda with
> reserved concurrency is the canonical pattern for rate-limiting FM calls.

## Task 4.3 -- Implement Monitoring Systems for GenAI Applications

### Key Concepts

**Holistic Observability Systems (Skill 4.3.1)**

GenAI observability extends the three traditional pillars (metrics, traces, logs) with
FM-specific signals: token consumption, response quality, and reasoning path visibility.

**CloudWatch Runtime Metrics**

Bedrock publishes metrics under `AWS/Bedrock`:

| Metric | Description |
|---|---|
| `Invocations` | Successful requests (Converse, InvokeModel, and streaming variants) |
| `InvocationLatency` | Time from request to last token (ms) |
| `InvocationClientErrors` | 4xx errors |
| `InvocationServerErrors` | 5xx errors |
| `InvocationThrottles` | Throttled requests (do NOT count as Invocations or Errors) |
| `InputTokenCount` | Input tokens |
| `OutputTokenCount` | Output tokens |
| `TimeToFirstToken` | Time to first token (streaming APIs only) |
| `EstimatedTPMQuotaUsage` | Quota consumption including cache writes and burndown multipliers |

The `ModelId` dimension filters all metrics by model. TTFT and EstimatedTPMQuotaUsage are
emitted at no additional cost, updated every minute.

**Bedrock Agent Traces**

Every agent response includes a structured trace (enable with `enableTrace: TRUE` in the
`InvokeAgent` request, disabled by default). Trace types:

- **PreProcessingTrace**: how the agent contextualizes and categorizes inputs.
- **OrchestrationTrace**: decision logic -- which actions to execute, which knowledge bases to
  query. Contains `rationale` (reasoning), `invocationInput` (action being invoked),
  `modelInvocationInput`/`Output`, and `observation` (results).
- **PostProcessingTrace**: how the agent shapes the final response.
- **FailureTrace**: reason a step failed.
- **GuardrailTrace**: actions taken by guardrails.
- **RoutingClassifierTrace**: input/output of multi-agent routing decisions.
- **CustomOrchestrationTrace**: for custom orchestration logic.

**X-Ray Integration**

- Bedrock AgentCore integrates with X-Ray via the **AWS Distro for OpenTelemetry (ADOT) SDK**.
- X-Ray SDKs are in maintenance mode; AWS recommends migrating to OpenTelemetry.
- AgentCore emits telemetry in **OpenTelemetry-compatible format**.

**Custom CloudWatch Dashboards**

Pre-configured dashboards include invocation count, latency, token counts by model, daily
trends, throttle counts. Build custom dashboards combining Bedrock, Agent, Guardrail, and
Knowledge Base metrics alongside infrastructure metrics.

**Business Impact Metrics**

These are custom metrics (not built-in). Examples: task completion rates, user satisfaction,
revenue per interaction, support deflection, cost per conversation. Use
`CloudWatch PutMetricData` or embed metrics via CloudWatch Logs metric filters.

> **Exam tip:** Throttled requests do NOT count as Invocations or Errors. TTFT is only available
> for streaming API operations. EstimatedTPMQuotaUsage accounts for cache write tokens and
> output burndown multipliers.

### Comprehensive GenAI Monitoring (Skill 4.3.2)

**Model Invocation Logging**

Must be explicitly enabled (disabled by default). Configuration via
`PutModelInvocationLoggingConfiguration` or the console Settings page.

Logs can be sent to:
- **CloudWatch Logs** -- for real-time analysis via Logs Insights.
- **Amazon S3** -- required for large data, binary content (images, video).
- Both simultaneously.

Captures: full request/response payloads, metadata (model ID, request ID, timestamp, token
counts, latency). Choose which modalities to log (text, image, embedding, video). The S3
bucket must be in the same Region as the Bedrock logging configuration.

**Prompt Effectiveness Monitoring with Bedrock Evaluations**

- **Programmatic evaluations**: computed metrics for accuracy, robustness, toxicity.
- **LLM-as-a-Judge**: a second LLM scores responses on correctness, completeness, helpfulness,
  logical coherence, faithfulness, harmfulness, stereotyping, answer refusal.
- **RAG evaluations**: context relevance, context coverage, citation precision, citation
  coverage, correctness, completeness, faithfulness.
- **Bring Your Own Inference (BYOI)**: evaluate responses from models hosted anywhere.
- Maximum **1,000 prompts per evaluation** job.

**Hallucination Rate Tracking**

Three mechanisms:

1. **Contextual Grounding Checks** (Bedrock Guardrails): validates responses against a grounding
   source and user query. Filters over 75% of hallucinated responses for RAG/summarization.
   Configurable thresholds per chunk.
2. **Automated Reasoning Checks** (Bedrock Guardrails): uses formal mathematical logic to verify
   accuracy. Up to 99% verification accuracy. Encodes domain rules as a policy. Limited regional
   availability.
3. **Custom detection**: build agentic workflows with Bedrock Agents that detect hallucinations
   and route to human review via SNS.

**Anomaly Detection**

*CloudWatch Anomaly Detection* (metrics):
- ML algorithms analyze past data and create models of expected values.
- Accounts for hourly, daily, and weekly patterns.
- Alarms trigger when metrics go above, below, or outside the expected band.
- Apply to `InputTokenCount`, `OutputTokenCount`, `InvocationLatency`.

*CloudWatch Logs Anomaly Detection*:
- `anomaly` command in Logs Insights identifies unusual log patterns.
- Assigns priority and severity levels.
- Can suppress anomalies temporarily or permanently.

**AWS Cost Anomaly Detection**

- ML-based identification of anomalous spend patterns.
- Runs approximately 3 times per day (not real-time).
- Monitor types: AWS Managed (auto-track all accounts/services) and Customer Managed (specific
  accounts/tags, max 10 per monitor).
- Alert via email, SNS, EventBridge, or AWS User Notifications.

> **Exam tip:** Model invocation logging is disabled by default. Contextual grounding checks
> require a grounding source -- they cannot operate without reference material. Automated
> Reasoning checks provide deterministic, mathematically provable verification (unique among
> GenAI safeguards). Cost Anomaly Detection is NOT real-time.

### Tool Performance Frameworks (Skill 4.3.4)

**Agent Metrics**

Bedrock Agents publish to the `AWS/Bedrock/Agents` namespace (not `AWS/Bedrock`):

| Metric | Description |
|---|---|
| `InvocationCount` | Requests to InvokeAgent/InvokeInlineAgent |
| `TotalTime` | End-to-end server processing time |
| `TTFT` | Time-to-first-token (streaming only) |
| `ModelLatency` | Time the FM took |
| `ModelInvocationCount` | Number of FM calls the agent made |
| `InputTokenCount` / `OutputTokenCount` | Token consumption |

Dimensions: `Operation`, `ModelId`, `AgentAliasArn`.

**OrchestrationTrace for Tool Analysis**

The trace reveals tool call patterns: which action groups were selected (`invocationInput`), why
(`rationale`), what the tool returned (`observation`), and how many FM calls were needed.

**External Tool Monitoring**

Lambda functions as action groups publish their own CloudWatch metrics (Duration, Errors,
Throttles, ConcurrentExecutions). Correlate Lambda metrics with Agent metrics via request IDs.
X-Ray tracing spans across agent-to-Lambda invocations.

**Multi-Agent Coordination Tracking**

- `RoutingClassifierTrace` traces routing classifier decisions for multi-agent setups.
- AgentCore supports Agent-to-Agent (A2A) protocol and Model Context Protocol (MCP).
- Track coordination through linked traces across agents.

**Usage Baselines**

Establish baselines and alarm on deviations:
- Number of FM calls per agent invocation (if normally 3 but suddenly 10, that's an anomaly).
- Token consumption per interaction.
- Tool selection frequency distributions.
- Error rates per tool.

> **Exam tip:** Agent metrics are in `AWS/Bedrock/Agents`, NOT `AWS/Bedrock`. The agent IAM
> service role needs explicit `cloudwatch:PutMetricData` permission for this namespace. TTFT
> for agents requires streaming configuration to be enabled.

### FM-Specific Troubleshooting Frameworks (Skill 4.3.6)

**Golden Datasets for Hallucination Detection**

A curated set of question-answer pairs with verified correct answers (ground truth):

- Bedrock Model Evaluation supports ground-truth datasets in **JSONL format** in S3.
- Maximum 1,000 prompts per evaluation.
- S3 bucket requires CORS permissions.
- Programmatic evaluations compare outputs against known-correct answers.
- LLM-as-a-Judge scores outputs for correctness, faithfulness, completeness.

**Output Diffing for Response Consistency**

- Run the same prompt multiple times and compare outputs.
- Track response variation across model versions.
- Use Bedrock Evaluations with BYOI to compare responses from different configurations.
- Store outputs in S3, query with Athena for systematic comparison.
- Use LLM-as-a-Judge to score consistency between outputs.

**Reasoning Path Tracing for Logical Errors**

The Agent Trace is the primary tool. Debugging workflow:
1. Enable traces (`enableTrace: TRUE`).
2. Examine PreProcessingTrace -- verify correct intent classification.
3. Walk through OrchestrationTrace steps -- find where reasoning diverges.
4. Check if correct knowledge bases and action groups were invoked.
5. Examine PostProcessingTrace -- verify response formatting.
6. Look for FailureTrace -- identify explicit error points.

AgentCore Observability provides end-to-end prompt tracing in CloudWatch for inspecting each
step, auditing intermediate outputs, and debugging bottlenecks.

**Specialized Observability Pipeline Architecture**

1. **Collection**: invocation logs (CloudWatch Logs + S3), agent traces, AgentCore telemetry
   (OpenTelemetry), CloudTrail audit events, custom metrics.
2. **Analysis**: Logs Insights for real-time queries, Anomaly Detection for metrics and logs,
   Athena for S3 log analysis, Bedrock Evaluations for quality.
3. **Visualization**: CloudWatch dashboards, Managed Grafana, QuickSight.
4. **Alerting**: CloudWatch Alarms, EventBridge rules, SNS notifications, Cost Anomaly Detection.
5. **Third-party integration**: Arize AI Phoenix, Datadog, Langfuse, IBM Instana -- all via
   OpenTelemetry.

> **Exam tip:** Golden datasets must be JSONL in S3 with CORS enabled. Agent traces are the
> primary tool for reasoning path analysis -- look for the `rationale` field. Automated Reasoning
> checks are the only GenAI safeguard with deterministic, mathematically provable verification.
> There is no single "hallucination metric" in CloudWatch.

## Common Exam Pitfalls for Domain 4

1. **Forgetting to enable logging**: Model invocation logging, knowledge base logging, and
   CloudTrail data events are all off by default.
2. **Wrong CloudWatch namespace**: Agent metrics are in `AWS/Bedrock/Agents`, not `AWS/Bedrock`.
3. **Confusing caching types**: Bedrock prompt caching (model computation state) vs. semantic
   caching (full response, bypasses LLM) vs. edge caching (CloudFront, deterministic responses).
4. **`max_tokens` and quota**: it is deducted at request start, not end. Setting it too high
   reduces concurrency even though billing is based on actual tokens.
5. **Burndown rate**: 5x for Claude 3.7+ output tokens. Affects quota, not billing.
6. **Streaming does not change cost**: same tokens billed regardless, only perceived latency
   improves.
7. **Bedrock manages its own scaling**: you do not configure auto-scaling for Bedrock on-demand
   inference. You do configure scaling for SageMaker endpoints and Lambda.
8. **Automated Reasoning vs. Contextual Grounding**: Automated Reasoning uses formal math
   (deterministic); Contextual Grounding uses probabilistic NLP assessment.
9. **Hybrid search prerequisite**: requires a filterable text field in the vector store.
10. **Cost Anomaly Detection timing**: runs ~3 times daily, not real-time.
